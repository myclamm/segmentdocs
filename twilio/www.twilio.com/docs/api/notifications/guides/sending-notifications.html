    Sending Notifications

    Now that we've registered the client app with GCM and created a Binding, it's time to send your first notification. Let's get started!

    Table of Contents

    • Server: Sending Notifications
    • Client: Receiving Notifications
    • Client: Notification Scenarios

    Server: Sending Notifications

    We'll want to send a POST request to Twilio from notify.js in our server app. The request should authenticate using your Twilio Account SID and Auth Token. It should also identify who to send a notification to by either their identity or a tag.

    We'll use identity if we want a notification to be sent to all devices with Bindings associated with a given identity.

    Send a Notification (identity)
    • curl
    • Java
    • Node.js
    // NOTE: This example uses the next generation Twilio helper library - for more
// information on how to download and install this version, visit
// https://www.twilio.com/docs/libraries/java
import com.twilio.Twilio;
import com.twilio.rest.notify.v1.service.Notification;

public class Example {
  // Find your Account Sid and Token at twilio.com/user/account
  public static final String ACCOUNT_SID = "ACCOUNT_SID";
  public static final String AUTH_TOKEN = "AUTH_TOKEN";

  public static final String SERVICE_SID = "ISXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX";

  public static void main(String[] args) {
    // Initialize the client
    Twilio.init(ACCOUNT_SID, AUTH_TOKEN);

    Notification notification = Notification
        .creator(SERVICE_SID)
        .setBody("Hello Bob")
        .setIdentity("Bob")
        .create();

    System.out.println(notification.getSid());
  }
}

    curl -X POST https://notify.twilio.com/v1/Services/ISxxx/Notifications \
    -d 'Identity=Bob' \
    -d 'Body=Hello Bob' \
    -u '{twilio account sid}:{twilio auth token}'

    // NOTE: This example uses the next generation Twilio helper library - for more
// information on how to download and install this version, visit
// https://www.twilio.com/docs/libraries/node
var env = require('./config.json');
var twilio = require('twilio');

// Authenticate with Twilio
var client = new twilio(env.TWILIO_ACCOUNT_SID,  env.TWILIO_AUTH_TOKEN);

// Create a reference to the user notification service
var service = client.notifications.v1.services(env.TWILIO_NOTIFICATION_SERVICE_SID);

// Send a notification 
service.notifications.create({
  'identity':'' + process.argv.slice(2),
  'body':'Hello, ' + process.argv.slice(2) + '!'
}).then(function(response) {
  console.log(response);
}).catch(function(error) {
  console.log(error);
});

    Client: Receiving Notifications

    The Beatles once sung 'Let it out and let it in', so open your heart (err, app) and let the notifications in. Both Android and iOS have native listener functions that fire whenever a notification is received.

    For iOS, this happens in our AppDelegate.

    Receiving a Notification in iOS
    • Objective-C
    • Swift
    -(void) application:(UIApplication *) application didReceiveRemoteNotification:(nonnull NSDictionary *)userInfo {
  NSLog(@"The notification message is: %@", [userInfo valueForKeyPath:@"aps.alert"]);
}

    func application(application: UIApplication, didReceiveRemoteNotification userInfo: [NSObject : AnyObject]) {
  let message = userInfo["aps"]?["alert"] as? String
  println(message)
}
    Receiving a Notification in iOS

    For Android, this happens in MyGcmListenerService.

    Receiving a Notification in Android
    • Java
    private static final String TAG = "MyGcmListenerService";
  @Override
  public void onMessageReceived(String from, Bundle data) {
     String body = data.getString("twi_body");
     Log.d(TAG, "From: " + from);
     Log.d(TAG, "Body: " + body);
  }
}
    Receiving a Notification in Android

    The notification will arrive in JSON format and contains the message body as a String that we can then use locally.

    Client: Notification Scenarios

    There are a number of ways we can potentially receive a notification depending on the state of our app and our device type. Let's take look at some of these scenarios.

    iOS

    Scenario 1: App is in the Background

    • If it's an alert notification, it will show on the home screen and in notification center. The notification can be retrieved by parsing the launch options dictionary in application:willFinish LaunchingWithOptions: and application:didFinishLaunchingWithOptions (more on this here).

    • If it's a silent notification, it will be delivered to application:didReceiveRemoteNotification :fetchCompletionHandler: and no alert will be raised (more on this here).

    Scenario 2: App is in the Foreground

    • The application:didReceiveRemoteNotification:fetchCompletionHandler: method is called but no alert will be raised.

    Android

    Scenario 1: App is in the Background

    • Data message (default): If there is only a data bundle then the app is woken up in the background and the onMessageReceived function is called. This is the default behavior in Notify as we map all parameters to the data bundle. Read more in our [REST API docs] (/docs/api/notifications/rest/notifications).

    • Notification message: If there is nothing in the data section, then it's added to the notification center.

    • Hybrid message: If there is a data section and a notification section, then it is added to the notification center and if the user clicks on the notification then the app is woken up and the data bundle is handed to the onMessageReceived function.

    Scenario 2: App is in the Foreground

    The notification is delivered to onMessageReceived of the GcmListenerService. The data key-value pairs can be found in the data bundle but the key-value pairs of the notification bundle are not available.

    Nuggety! Notifications now notify nicely. No nitpicking, no nagging! Need novelty? Nibble on our REST API Guide. It'll give you a more in-depth look at the Notifications resources.
